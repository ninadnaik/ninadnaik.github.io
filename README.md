# Ninad Naik — Personal Website

Source for **https://ninadnaik.com** — a Hugo static site (PaperMod theme) that
auto-deploys to GitHub Pages. This file is maintainer documentation; it is **not**
part of the site build (Hugo only renders `content/`, so this README never ships).

---

## What's in this repo

This is really two things living side by side:

```
┌────────────────────────────┐        ┌────────────────────────────────┐
│  1. The website            │        │  2. Research triage automation │
│  Hugo + PaperMod theme     │        │  Weekly GitHub Action          │
│  → GitHub Pages → .com      │        │  → drafts into research/       │
└────────────────────────────┘        └────────────────────────────────┘
        (works today)                       (currently broken — see below)
```

## 1. The website

### Stack

| Piece        | Detail                                                        |
|--------------|---------------------------------------------------------------|
| Generator    | [Hugo](https://gohugo.io) extended — CI pins **v0.139.0**     |
| Theme        | [PaperMod](https://github.com/adityatelange/hugo-PaperMod) — a **git submodule** at `themes/PaperMod` |
| Config       | `hugo.toml`                                                   |
| Hosting      | GitHub Pages, custom domain via `CNAME` (`ninadnaik.com`)     |
| Deploy       | `.github/workflows/hugo.yml` on every push to `main`          |

### Run it locally

The theme is a submodule, so a fresh clone needs it pulled in:

```bash
git clone --recurse-submodules https://github.com/ninadnaik/ninadnaik.github.io.git
# already cloned without it?
git submodule update --init --recursive

hugo server -D       # live preview at http://localhost:1313 (-D includes drafts)
```

### Build / deploy flow

```
  edit content/  ──► git push origin main ──► GitHub Action (hugo.yml)
                                                    │
                                       hugo --gc --minify  (builds ./public)
                                                    │
                                          deploy to GitHub Pages
                                                    │
                                            live at ninadnaik.com
```

There is no manual build or deploy step — **push to `main` is publish.** `public/`
is generated by CI and is gitignored; never commit it.

## 2. Writing & publishing a post

Posts are **page bundles**: each post is its own folder under `content/posts/`
containing an `index.md`, plus any images/SVGs it references.

### Folder-name convention

The prefix signals the post's category (it does **not** appear in the URL):

| Prefix    | Meaning            | Example                          |
|-----------|--------------------|----------------------------------|
| `0N_`     | Intelligence Series| `01_intelligence/`               |
| `dd_`     | Deep Dive          | `dd_llms-io-layer/`              |
| `macro_`  | Macro essay        | `macro_trust-regimes/`           |

### Frontmatter (YAML)

Every post starts with this block. The `slug` — not the folder name — sets the URL
(`/posts/<slug>/`), and the leading category label (e.g. `Macro:`, `Deep Dive:`) is
dropped from the slug.

```yaml
---
title: "Macro: Trust, Regimes, and How to Position Across Them"
slug: "trust-regimes-and-how-to-position"
date: 2026-06-17
tags: ["Macro", "Investing", "Trust", "Economics"]
author: "Ninad Naik"
description: "One- to two-sentence summary shown in listings and meta tags."
---
```

### Checklist

1. Create `content/posts/<prefix>_<short-name>/index.md`.
2. Add the frontmatter above; pick a concise kebab-case `slug`.
3. Drop any diagrams (`.svg`/images) in the same folder; reference them relatively.
4. Preview: `hugo server -D`.
5. Sanity-build: `hugo --gc` (catches frontmatter/markdown errors).
6. `git add` the post folder, commit (`Publish <Title>`), `git push` → live in ~1 min.

## 4. Analytics

The site uses **Cloudflare Web Analytics** — privacy-friendly, cookieless, and
therefore **no consent banner required**. It works on GitHub Pages without proxying
the domain through Cloudflare (it's just a JS beacon).

### How it's wired

```
PaperMod head.html ──► partial "extend_head.html" ──► Cloudflare beacon
                       (our project-level override)    (production builds only)
```

- The snippet lives in **`layouts/partials/extend_head.html`**, a project override of
  the theme's empty `extend_head.html` — so a PaperMod submodule update can't wipe it.
- It's gated on **`hugo.IsProduction`**, so the beacon fires only on real `hugo`
  builds (i.e. `ninadnaik.com`) and **never during `hugo server` previews** — your
  local browsing isn't counted.
  - Note: it must use `hugo.IsProduction`, *not* `site.Params.env`, because
    `hugo.toml` hardcodes `env = "production"` and so can't distinguish dev from prod.
- The beacon **token** is a public client-side identifier (safe to commit, lives
  inline in the snippet). Not a secret — no `.env` involved.

### Where to see the data

Cloudflare dashboard → **Analytics & Logs → Web Analytics** (site: `ninadnaik.com`).
Data appears within minutes of real (non-preview) traffic.

### Rotating or replacing it

Get a new beacon token from the Cloudflare dashboard and swap the `data-cf-beacon`
token in `layouts/partials/extend_head.html`. To switch providers entirely, replace
the snippet in that same file.

## Conventions & gotchas

- **Push = publish.** There is no staging environment; `main` is production.
- **Don't edit `themes/PaperMod/`** — it's an upstream submodule. Override via
  `layouts/` or `assets/` instead.
- **Don't commit `public/`** — it's CI-generated and gitignored.
- **Local Hugo should match CI** (`v0.139.0 extended`) to avoid surprises.
- **Analytics won't show in local previews** — the beacon is production-only by
  design, so `hugo server` pages carry no tracking (see section 4).
